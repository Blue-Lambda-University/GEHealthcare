1. Purpose and Scope

This document describes the architectural design for Custom Option 2: a custom-built GenAI solution that orchestrates SDLC agents (Industry Align → User Story → Windchill Align → FuncSpec → TestCaseGen → DesignDoc → PseudoCode) with human-in-the-loop gates, integrated with JIRA and Windchill. It is intended for solution architects, technical leads, and implementers. It defines structure, components, interfaces, data flows, and key design decisions; it does not specify implementation code.

In scope: Logical and deployment views of the system; component responsibilities and interfaces; integration via MCP and external systems; security and governance posture.

Out of scope: Detailed API specs, database schemas, or line-by-line implementation.

4. Architectural Goals and Constraints

4.1 Goals

  • Vendor independence: Ability to swap LLM providers and add/replace agents or tools without rewriting the core pipeline.
  • Clear separation of concerns: Orchestration, agent logic, tool integration, and knowledge are separate; agents do not call external systems directly.
  • Human-in-the-loop: Two explicit approval gates (Product Owner, Solution Architect) with audit trail.
  • Extensibility: New agents, tools, or knowledge sources can be added via configuration or new MCP servers.
  • Observability: Runs are traceable; token usage and failures can be monitored for cost and reliability.

4.2 Constraints

  • No direct agent–system calls: Agents receive and return data only via the Orchestrator; all JIRA/Windchill/file access goes through the Tools layer (MCP).
  • Enterprise security: Identity via enterprise SSO; secrets in secure store; data in transit and at rest encrypted; processing in approved regions.
  • Compliance: Audit log of runs, approvals, and tool invocations; alignment with GE HealthCare data and AI policies.

6. Component Design

6.1 Front End Application

Responsibility: Single entry point for users to trigger pipeline runs, view run status and agent outputs, and perform Product Owner and Solution Architect reviews.

Interfaces (conceptual):

  • To Orchestrator: Start run (source: “user needs list” or “JIRA project/epic”); get run status; get step output; submit review decision; get run history.
  • From Orchestrator: Run created; step completed; gate pending; run completed / failed.

Design notes: Authenticate via enterprise SSO; authorize by role. For human gates, front end displays artifacts to review and sends approve/reject/edits back.

6.2 GenAI Orchestrator

Responsibility: Execute the SDLC pipeline; pass data between agents and tools; enforce human-in-the-loop gates; persist run state and audit events.

Pipeline definition (logical): 1) Load input (via MCP). 2) Industry Align → 3) User Story → 4) Windchill Align (update JIRA OOTB fitment) → 5) FuncSpec → 6) TestCaseGen → 7) Gate 1 (PO) → 8) DesignDoc → 9) Gate 2 (SA) → 10) PseudoCode → 11) Update JIRA → 12) Optional export.

Interfaces: To Front End (start run, status, step output, review decision, history); to Agents (invoke with payload, receive output); to MCP Client (list/call tools).

6.3 Agents

Responsibility: One specialist role each; receive input from Orchestrator; call LLM (and optionally Knowledge); return structured output. Agents do not call JIRA, Windchill, or file systems directly.

AgentInputOutput
Industry AlignRequirementsRequirement, Meets Industry Practice, Rationale, Alignment Actions
User StoryRequirements + alignmentEpics, user stories (Actor, Description, Prerequisites, Steps, Business rules, etc.)
Windchill AlignUser storiesFitment assessment; Orchestrator writes OOTB fitment via JIRA MCP
FuncSpecUser stories + fitmentFunctional specification document
TestCaseGenUser stories / functional specTest cases (agreed format)
DesignDocFunctional spec + contextSystem overview, HLD, LLD, design decisions
PseudoCodeUser stories (customization scope)Structured pseudocode by epic/feature

6.4 MCP Client

Connect to MCP servers; discover tools (tools/list); invoke tools (tools/call); return results to Orchestrator. Credentials in secure store.

6.5 MCP Servers (Tools and Knowledge)

JIRA MCP Server: Tools e.g. get_issues, get_requirements, create_or_update_issue, update_custom_fields. Backing: JIRA REST API.

Windchill MCP Server: Tools e.g. get_solution_metadata, get_ootb_capabilities, evaluate_fitment. Backing: Windchill API or thin wrapper.

File/Document MCP Server: Tools e.g. read_file, write_artifact, list_templates, export_to_path.

Knowledge MCP Server: Tools/resources e.g. search_standards, get_template, get_playbook or RAG.

9. Security and Governance

9.1 Authentication and Authorization

Front End: enterprise SSO (SAML/OIDC). Orchestrator API: token or API key; authorize by role. MCP servers: TLS; per-server credentials in vault. Agents: invoked by Orchestrator only; LLM API key from vault.

9.2 Data Protection

TLS in transit; encryption at rest for persisted state and logs; role-based access to run outputs; audit of access.

9.3 Audit and Compliance

Orchestrator logs: run ID, user/trigger, start/end time, each step and gate, tool invocations. Retention per GE HealthCare policy. Human gates ensure no automatic publish of high-impact artifacts.

9.4 Vendor Neutrality (LLM)

Agent calls an internal LLM gateway/adapter; provider and model configurable by environment or tenant.

10. Key Design Decisions

DecisionRationale
Tools exposed via MCPStandard protocol for discovery and invocation; easy to add new integrations; aligns with ecosystem.
Agents do not call external systemsClear separation; agents swappable and testable; integration and security in MCP servers and Orchestrator.
Orchestrator owns pipeline and gatesSingle place for sequence, retries, and human gates; front end only displays and submits decisions.
Two human gates (PO, SA)Matches GE HealthCare SDLC and ensures accountability before user stories/test cases and before design/pseudocode.
Structured payloads per stepEnables versioning of agent contracts and backward compatibility.
LLM abstractionEnables swap of provider without changing agent implementations.
