1. Purpose and Scope

This document describes the architectural design for the ServiceNow + LLM warehouse incident solution: how components are structured, how data flows from incident creation through policy rules and LLM to ServiceNow updates, and how external systems (OTM, xCarrier, MSCA, logs, Knowledge) integrate. It is intended for solution architects and implementers. It does not specify implementation code.

In scope: System context, logical architecture, component responsibilities and interfaces, data flows, security and governance, key design decisions.

Out of scope: Detailed API specs, database DDL, or prompt text.

3. Architectural Goals and Constraints

3.1 Goals

  • ServiceNow as system of record: All incident state and AI outputs live in ServiceNow; no duplicate system of record.
  • Policy-first: Deterministic rules (warehouse min P2, ops halted → P1, 2hr monitor, security → P1) run first and can override or bound LLM output.
  • Enriched context: LLM receives incident + CMDB + related + knowledge base + warehouse health snapshot + log excerpts.
  • Auditability: Full audit of what was pulled, what rules fired, what the LLM returned, and who approved or overrode.
  • Safety: No automatic downgrade of P1 without approval; optional human approval for all P1 or for LLM-suggested P1 when rules did not set P1.

3.2 Constraints

  • No direct LLM write to production incident priority without guardrails (rules override or human approval).
  • PHI/PII: Redact in logs and in any stored payload before sending to LLM or logging.
  • Token and cost: Enriched payload must be size-limited (truncation, snippet caps).

5. Component Design

5.1 Orchestrator

Responsibility: Single entry point for both incident intake and 2hr monitor flows. Coordinates context aggregation, policy execution, LLM call, and write-back.

Flow (intake): Receive trigger → Call Context Aggregator → Call Policy Engine → Build LLM payload → Call LLM Gateway → Apply guardrails → Call ServiceNow Adapter to write work notes, u_ai_*, optionally priority → Set 2hr monitor if P2 warehouse; notify resolver group.

Flow (2hr monitor): Receive list of incident IDs → For each: refresh context and policy → If impact grown or not resolved → escalate to P1, update ServiceNow, trigger P1 automation → Post automated work note.

Interfaces: Input: incident_id, trigger_type (intake | 2hr_monitor). Output: success/failure; optional callback for escalate to human.

5.2 Policy / Rules Engine

Responsibility: Pure deterministic logic. No LLM. Input: structured fields (site, warehouse_type, ops_state, stations_impacted_count, workaround_present, workaround_quality, security_compliance_risk). Output: applied_priority (P1|P2|P3), reason, rule_triggered, start_2hr_monitor.

Rules: (1) security_compliance_risk → P1. (2) operations_halted OR stations_impacted >= threshold → P1. (3) services_warehouse AND below threshold → min P2; start_2hr_monitor = true. (4) Else → standard ITIL or pass-through.

5.3 Context Aggregator

Assembles all context for one incident: calls ServiceNow Adapter, Warehouse Signals Adapter, Log Service, Knowledge Retriever; normalizes into payload shape; applies time window and token/size caps.

Interface: Input: incident_id, options. Output: enriched payload.

5.4 LLM Gateway

Single point of call to LLM provider. Builds prompt with policy summary and enriched payload; requests structured JSON (priority, confidence, justification, suggested_actions, missing_information); validates response; redacts PHI/PII as needed.

5.5–5.8 Adapters and Services

ServiceNow Adapter: Read incident, CMDB, related, assignment, Knowledge; write u_ai_*, work notes, optionally priority. Uses ServiceNow REST or mid-server; credentials in vault.

Warehouse Signals Adapter: Call OTM, xCarrier, MSCA (or aggregated API); map to warehouse health snapshot schema.

Log Service: Query log aggregation by time, site, CI, app; return truncated excerpts; redact PHI/PII; return empty or "Logs unavailable" on failure.

Knowledge Retriever: Query ServiceNow Knowledge by category, CI, or search; return list of { title, snippet } with length cap.

6.2 2-Hour Warehouse Monitor (Flow 2)

Orchestrator receives incident IDs from scheduler. For each: refresh context and policy; if impact grown or still not resolved → escalate to P1, update ServiceNow, trigger P1 automation; post automated work note.

6.3 P1 Major Incident Automation (Flow 3)

Triggered when priority = P1. Steps: open bridge, notify on-call, notify leadership, start update cadence, maintain executive summary. May be in ServiceNow or via Orchestrator.

ServiceNow remains in its cloud tenant. Orchestrator, Policy Engine, Context Aggregator, LLM Gateway can run in one or more services (e.g. cloud function, container). Adapters and Log Service need network access to ServiceNow, OTM/xCarrier/MSCA, and log aggregation. Secrets in vault; no secrets in code.

8. Security and Governance

  • Authentication: ServiceNow adapter uses OAuth or API token (vault). Warehouse, log, and LLM use respective credentials (vault).
  • Authorization: Service identity with least privilege (read incident/CMDB/Knowledge, write incident/work notes) in ServiceNow.
  • Data: PHI/PII redaction in logs and payloads before LLM or audit log.
  • Audit: Log for each run: incident_id, trigger_type, policy_result, LLM input summary (no PHI), LLM output, final priority, who approved/overrode. Retain per GE HealthCare policy.
  • No auto-downgrade of P1 without explicit approval (configurable guardrail).

9. Key Design Decisions

DecisionRationale
ServiceNow as system of recordSingle source of truth; no sync of incident state elsewhere.
Policy engine runs before LLMDeterministic rules enforce safety and warehouse policy; LLM augments within bounds.
Enriched payload (incident + snapshot + logs + knowledge)LLM can summarize and justify with evidence; better drafts and next steps.
Structured LLM output (JSON)Enables automation and guardrails; validation prevents invalid priority.
Optional human-in-the-loop for P1 / downgradeAligns with Incident Priority Definitions; reduces misclassification risk.
Separate adapters for ServiceNow, warehouse, logs, knowledgeClear boundaries; each can be replaced or scaled independently.
